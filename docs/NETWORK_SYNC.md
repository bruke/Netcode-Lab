# 网络同步学习：帧同步与状态同步

本仓库用 **Node.js（WebSocket）** 做服务器，**两个独立静态页面** 分别演示 **帧同步（Frame Sync）** 与 **状态同步（State Sync）** 的典型数据流。运行方式见文末。

---

## 核心概念对照

| 维度 | 帧同步（本 demo） | 状态同步（本 demo） |
|------|-------------------|---------------------|
| 服务器每拍发什么 | 各玩家在本逻辑帧的 **输入**（按键布尔） | 各玩家的 **权威位置**（坐标） |
| 客户端如何得到画面 | 用 **相同规则** 在本地按帧推进（确定性推演） | **直接绘制** 服务器下发的状态 |
| 权威在谁 | 逻辑上「输入流 + 规则」一致即一致；服务器仍可做镜像推演用于校验/断线重连 | **位置以服务器为准** |
| 典型代价 | 需保证确定性、延迟与输入对齐；复杂玩法要处理不同步与回滚 | 状态带宽随实体数上升；常配合预测与插值 |

两者不是互斥标签：商业项目里常混合使用（例如帧同步式输入 + 低频状态校验）。

---

## 本项目的协议说明

公共约定：

- 逻辑节拍 **20 TPS**（`TICK_MS = 50`），与 `server/src/constants.js` 及前端常量一致。
- 地图为 **400×400** 平面，方块 **28px**，每拍位移 **5px**（与方向键组合有关）。

### 帧同步：`/ws/frame-sync`

1. **welcome**：`{ type, mode, id, frame, tickMs, players: [{ id, x, y }] }`  
   连接后立即下发，用于对齐初始位置与当前逻辑帧号。
2. **input（客户端 → 服务器）**：`{ type: 'input', keys: { up, down, left, right } }`  
   客户端以较高频率（本 demo 约 30Hz）上报当前按键，服务器在 **下一拍** 使用「该拍收集到的输入」进入逻辑帧。
3. **frame（服务器 → 全体）**：`{ type: 'frame', frame, inputs: { [playerId]: keys } }`  
   每个逻辑帧广播 **所有玩家** 在本帧使用的输入；客户端用与服务器相同的 `stepPlayer` 规则本地推进，保证与服务器镜像一致。
4. **player_joined / player_left**：加入、离开时同步实体列表。

学习要点：客户端 **不依赖** 每帧接收坐标也能动（本 demo 中坐标由输入推演）；若要与服务器完全一致，必须 **初始状态一致 + 每帧输入序列一致 + 纯确定性运算**。

### 状态同步：`/ws/state-sync`

1. **welcome**：同上，含全体当前位置。
2. **input（客户端 → 服务器）**：同上，仅把按键交给服务器。
3. **state（服务器 → 全体）**：`{ type: 'state', frame, players: [{ id, x, y }] }`  
   服务器在节拍上积分按键更新位置，再 **广播坐标**；本 demo 的网页 **不做客户端预测**，你看到的就是「服务器说在哪就在哪」。

学习要点：实现简单、易于做复杂物理；但网络延迟会直接体现为 **操控滞后**，工业界常用 **客户端预测 + 服务器校正 + 插值** 优化手感（本 demo 未实现，便于观察原始差异）。

## 如何运行

在项目根目录：

```bash
npm install
npm start
```

浏览器访问：

- 导航页：<http://localhost:3000/>
- 帧同步：<http://localhost:3000/frame-sync/>
- 状态同步：<http://localhost:3000/state-sync/>

多开 **同一浏览器的多个标签** 或 **不同浏览器**，即可看到多个玩家 id；本机方块有白色描边。

## 建议的实验步骤

1. **帧同步页**：开两个标签，观察双方移动是否一致；可在开发者工具里查看 WebSocket 帧，对比 `frame.inputs` 与本地推演。
2. **状态同步页**：同样双开，感受 **仅发输入、只靠 state 更新位置** 的延迟感（本 demo 未做预测）。
3. 尝试把 `TICK_MS` 改大（如 100ms），观察两种模式下体验变化。
4. 阅读 `server/src/sim.js` 与 `client/frame-sync/app.js` 中的同名步进逻辑，理解 **「规则一致」** 在帧同步中的重要性。

## 延伸阅读（概念）

- **帧同步 / Lockstep**：常见于 RTS、部分格斗与怀旧联机；需处理浮点确定性、掉线、追帧等。
- **状态同步**：常见于 FPS、MOBA 等；实体多时需注意带宽与快照压缩。
- **输入延迟、缓冲、回滚式网络模型（如 GGPO）**：在帧同步框架上缓解延迟的高级话题，本 demo 未覆盖。

若你后续要在本仓库加「预测、插值、校验和、断线重连」，可以在现有协议上增量扩展。
